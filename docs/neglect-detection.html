Neglect Detection

The headline feature of Devansh OS. It continuously asks one question across every tracked system in your life — "What am I letting slip?" — and surfaces the answer, worst-first, as a banner at the top of the dashboard.

One-line summary: a generic engine walks every enabled provider, evaluates the neglect rules that provider declares against your daily metric history, and returns a severity-sorted list of warnings. Providers own what counts as neglect; the engine only knows how to measure it.

The core idea: recency over scores

Devansh OS is built around systems, not scores. It doesn't grade you on totals; it watches for silence. A streak you stopped defending, a deploy that hasn't shipped, an average that quietly drifted below a healthy floor — those are the failure modes the engine is tuned to catch. Everything below is in service of detecting that silence reliably and ranking it by how much it should worry you.

Architecture at a glance

Providers declare neglect_rules → neglect.evaluate() → aggregates over metric_daily → /api/neglect → banner (neglect.js)

The design deliberately keeps the engine generic. A new data source becomes neglect-aware by adding a single Rule to its provider class — no changes to the engine, the API, or the frontend.

The rule contract

A Rule is a static declaration owned by the provider that knows the metric. app/models.py

@dataclass
class Rule:
    metric: str                    # which metric to watch, e.g. "commits"
    kind: Literal["days_since", "rolling_avg_below"]
    label: str                     # human label shown in the banner ("Gym")
    warn: float | None = None      # days_since: warn threshold (days)
    crit: float | None = None      # days_since: critical threshold (days)
    window: int | None = None      # rolling_avg_below: window in days
    threshold: float | None = None # rolling_avg_below: avg below this triggers
    severity: Severity = "warning" # severity for rolling_avg_below
    unit: str = ""

There are exactly two ways the engine knows how to measure neglect, captured by kind.

How neglect is measured

1. days_since — defending a streak

The dominant rule type. It answers "how long since I last did this?" and escalates as the gap widens. app/neglect.py · _eval_days_since

The measurement comes from aggregates.days_since(), which builds on last_active_day():

  • It pulls the metric's daily series for roughly the last year (series_for_year → a 370-day window).
  • The last active day is the most recent day with a positive value (value > 0). Zero or missing days don't count as activity.
  • days_since = today − last_active_day, in whole calendar days (timezone-aware, via the configured tz).

The engine then maps that integer to a severity:

ConditionResult
days_since is None (metric never recorded)info — "…: no activity recorded yet", flagged once so the system still surfaces
days_since == 0 (active today)no warning — suppressed entirely
days_since ≥ critcritical
days_since ≥ warn (but below crit)warning
otherwise (within tolerance)no warning

The message is humanized: 1 day reads "yesterday", anything more reads "N days ago" — e.g. "Gym: last activity 4 days ago". The value carried alongside is the raw day count, which also drives tie-breaking in the sort.

Note the asymmetry: a metric that was never recorded is only info, not critical. The engine treats "you've never tracked this" as a setup gap to surface gently, while "you tracked this and then stopped" is the real neglect signal that escalates.

2. rolling_avg_below — guarding a floor

For metrics where the concern isn't "did I do it recently" but "is the sustained level healthy" — sleep being the canonical case. app/neglect.py · _eval_rolling_avg

It uses aggregates.rolling_avg(provider, metric, window), which is deliberately different from the streak math:

def rolling_avg(provider, metric, window):
    s = window_series(provider, metric, window)
    total = sum(s.values())
    return round(total / window, 2)   # divides by the FULL window

Crucially it divides by the entire window, including zero and missing days — so skipped days drag the average down rather than being ignored. (Contrast day_avg, used by cards, which averages only active days.) If the resulting average is below threshold, a warning is emitted at the rule's declared severity, with a message like "Sleep: 7-day average 5.4h (below 6h)".

The evaluation loop

evaluate() is the whole engine. It is intentionally tiny and provider-agnostic: app/neglect.py

def evaluate() -> list[dict]:
    warnings = []
    for provider in registry.enabled():        # only enabled providers
        for rule in provider.neglect_rules:     # each declared rule
            if rule.kind == "days_since":
                w = _eval_days_since(provider.key, rule.label, rule)
            elif rule.kind == "rolling_avg_below":
                w = _eval_rolling_avg(provider.key, rule.label, rule)
            else:
                w = None
            if w:
                warnings.append(w)
    warnings.sort(key=lambda w: (
        _SEVERITY_ORDER.get(w["severity"], 9),
        -(w["value"] or 0) if isinstance(w["value"], (int, float)) else 0))
    return warnings
  • Only enabled providers are evaluated. registry.enabled() filters out providers that lack their requirements (missing credentials, a disabled data source). A provider whose enabled() returns False — e.g. Sleep and ChemVecto, which are awaiting Phase 2 data sources — contributes no warnings at all.
  • Each warning is a plain dict: provider, label, severity, message, value. That keeps it trivially JSON-serializable for the API.

Sort order: worst-first

The list is sorted by a two-key tuple:

  1. Severity, using _SEVERITY_ORDER = {critical: 0, warning: 1, info: 2} — critical floats to the top.
  2. Within a severity, by value descending (the -(value)). For days_since rules this means the longest-neglected item leads; the thing you've ignored for 9 days outranks the one you've ignored for 4.

This ordering is the product: the most urgent neglect is always the first thing you read.

The HTTP surface

GET /api/neglect wraps evaluate() and adds the summary the banner needs. app/api/neglect.py

{
  "warnings": [ { provider, label, severity, message, value }, ... ],
  "counts":   { "critical": 1, "warning": 2, "info": 0 },
  "worst":    "critical"   // critical > warning > info > "ok"
}

counts tallies each severity; worst collapses the whole system into a single status, falling through critical → warning → info → "ok". When nothing fires, worst is "ok" and warnings is empty.

How it renders

The banner (web/static/js/neglect.js) reflects the engine's output directly:

  • The header shows the count chips (N critical N warning N info) and, when clear, all systems active.
  • container.dataset.worst is set to data.worst, so CSS can color the whole banner by overall status.
  • Each warning becomes a clickable row with a severity icon (✕ / ! / i) and the message; clicking calls back with the provider key to open that system's drill-down.
  • Empty state: "Nothing neglected — every tracked system has recent activity."

The Rituals provider is the one place rules aren't a static list. refresh_metrics() rebuilds neglect_rules from the active ritual bank whenever a ritual is added, edited, or removed, so the engine stays in sync. app/providers/rituals.py

self.neglect_rules = [
    Rule(metric=f"r_{r['id']}", kind="days_since", label=r["name"],

Each ritual derives its own thresholds from its interval_days: you get a warning as soon as you're past the cadence you set, and it escalates to critical at twice that interval. A ritual you intend to do weekly warns at 7 days and goes critical at 14.

End-to-end example

Suppose today is a day where you last committed 6 days ago, last hit the gym 4 days ago, and have never connected LeetCode:

  1. GitHub's commits rule: days_since = 6 ≥ crit (5) → critical, value 6.
  2. Gym's workout rule: days_since = 4, which is ≥ warn (3) but < crit (6) → warning, value 4.
  3. LeetCode's solved rule: never recorded → info, "no activity recorded yet".

After the sort (severity, then value desc), the banner reads:

2 critical?  no — counts: { critical: 1, warning: 1, info: 1 }, worst: "critical"

✕  GitHub: last activity 6 days ago
!  Gym: last activity 4 days ago
i  LeetCode: no activity recorded yet

Design notes & edge cases worth knowing

  • "Active" means positive. A logged 0 is treated as inactivity for days_since — only value > 0 resets a streak.
  • The lookback is ~1 year. last_active_day searches a 370-day window; activity older than that reads as "never recorded" (info). In practice this only matters for systems untouched for a year.
  • Same-day activity silences the alert. days_since == 0 returns no warning, so doing the thing today immediately clears it.
  • Missing days hurt the average. rolling_avg_below divides by the full window, so a week with skipped nights pulls the sleep average down — by design.
  • Enabled-gating is the safety valve. Providers without a data source (Sleep, ChemVecto) declare rules but stay silent until enabled, so the banner never nags about systems you haven't wired up.
  • The engine never crashes the dashboard. It only reads pre-aggregated metric_daily data; the heavy lifting (and any failure isolation) happens earlier, during each provider's sync().
